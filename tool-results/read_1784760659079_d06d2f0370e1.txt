  1515→/// Learning Engine — records when a human employee overrides the matcher's
  1516→/// chosen candidate (e.g. matcher picked candidates[0] but employee confirmed
  1517→/// candidates[1]). These overrides feed back into the `historicalMatch` evidence
  1518→/// signal so the matcher LEARNS from human corrections over time.
  1519→///
  1520→/// Lifecycle:
  1521→///   1. matcher returns candidates[0] as winner (productId=A)
  1522→///   2. employee reviews, decides productId=B (candidates[1]) is correct
  1523→///   3. POST /api/products/match-override records:
  1524→///        inputText, inputNormalized, fromProductId=A, toProductId=B,
  1525→///        chosenAlias, auditId (link to original ProductMatchAudit decision)
  1526→///   4. Next time the SAME normalized input is matched, the matcher sees a past
  1527→///      override and boosts confidence for productId=B's aliases.
  1528→///
  1529→/// `inputNormalized` is the Arabic-normalized form (via normalizeArabic) so that
  1530→/// minor spelling variations still hit the same override record. The compound
  1531→/// index `(companySlug, inputNormalized)` is the primary lookup path.
  1532→model MatchOverride {
  1533→  id              Int      @id @default(autoincrement())
  1534→  companySlug     String
  1535→  inputText       String // raw invoice line description that was overridden

  1537→  fromProductId   Int? // matcher's original winner (null if matcher said "new")
  1538→  toProductId     Int // product the employee chose instead
  1539→  chosenAlias     String? // alias text the employee confirmed
  1540→  auditId         Int? // link to ProductMatchAudit row of the original decision
  1541→  reason          String? // optional employee note ("brand mismatch", "wrong size")
