  1596→/// Learning Engine — records when a human employee overrides the matcher's
  1597→/// chosen candidate (e.g. matcher picked candidates[0] but employee confirmed
  1598→/// candidates[1]). These overrides feed back into the `historicalMatch` evidence
  1599→/// signal so the matcher LEARNS from human corrections over time.
  1600→///
  1601→/// Lifecycle:
  1602→///   1. matcher returns candidates[0] as winner (productId=A)
  1603→///   2. employee reviews, decides productId=B (candidates[1]) is correct
  1604→///   3. POST /api/products/match-override records:
  1605→///        inputText, inputNormalized, fromProductId=A, toProductId=B,
  1606→///        chosenAlias, auditId (link to original ProductMatchAudit decision)
  1607→///   4. Next time the SAME normalized input is matched, the matcher sees a past
  1608→///      override and boosts confidence for productId=B's aliases.
  1609→///
  1610→/// `inputNormalized` is the Arabic-normalized form (via normalizeArabic) so that
  1611→/// minor spelling variations still hit the same override record. The compound
  1612→/// index `(companySlug, inputNormalized)` is the primary lookup path.
  1613→model MatchOverride {
  1614→  id              Int      @id @default(autoincrement())

  1618→  fromProductId   Int? // matcher's original winner (null if matcher said "new")
  1619→  toProductId     Int // product the employee chose instead
  1620→  chosenAlias     String? // alias text the employee confirmed
  1621→  auditId         Int? // link to ProductMatchAudit row of the original decision
  1622→  reason          String? // optional employee note ("brand mismatch", "wrong size")
  1623→  overriddenBy    String // user email/id
