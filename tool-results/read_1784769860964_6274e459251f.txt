  1594→/// Learning Engine — records when a human employee overrides the matcher's
  1595→/// chosen candidate (e.g. matcher picked candidates[0] but employee confirmed
  1596→/// candidates[1]). These overrides feed back into the `historicalMatch` evidence
  1597→/// signal so the matcher LEARNS from human corrections over time.
  1598→///
  1599→/// Lifecycle:
  1600→///   1. matcher returns candidates[0] as winner (productId=A)
  1601→///   2. employee reviews, decides productId=B (candidates[1]) is correct
  1602→///   3. POST /api/products/match-override records:
  1603→///        inputText, inputNormalized, fromProductId=A, toProductId=B,
  1604→///        chosenAlias, auditId (link to original ProductMatchAudit decision)
  1605→///   4. Next time the SAME normalized input is matched, the matcher sees a past
  1606→///      override and boosts confidence for productId=B's aliases.
  1607→///
  1608→/// `inputNormalized` is the Arabic-normalized form (via normalizeArabic) so that
  1609→/// minor spelling variations still hit the same override record. The compound

  1662→// ────────────────────────────────────────────────────────────────────────────
  1663→// AI Orchestration Layer — Model Registry + Benchmark Results
  1664→//
  1665→// Replaces the "hardcoded provider chain" with a live, self-tuning registry.
  1666→// Each model declares which capabilities it supports (chat, invoice-extraction,
  1667→// reasoning, vision). A periodic auto-benchmark measures success rate, latency,
  1668→// and response quality per (model × capability), then rolls those metrics into
